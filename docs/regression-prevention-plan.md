# LEWORD 회귀 방지 플랜

## 회귀로 본 증상

- 황금키워드 발굴에서 카테고리만 선택하면 시드가 충분히 주입되지 않아 정밀 모드가 30개 미만으로 끝난다.
- PRO 트래픽 키워드 헌터에서 카테고리 모드 후보를 찾았어도 후처리 게이트가 지나치게 강해 결과가 10~20개대로 줄어든다.
- UI에는 "주의 마커를 붙여 넓게 보여준다"라고 안내하지만 실제 코드는 watchlist 후보를 차단하는 경로가 있었다.
- 패션/뷰티, 쇼핑, 여행, 반려동물처럼 UI 카테고리와 내부 카테고리 ID가 1:1이 아닌 항목에서 발굴 풀이 좁아졌다.
- 마인드맵/LEWORD 분석 검색량은 실측 매칭 실패를 0으로 보이게 하면 사용자 입장에서는 "정확하지 않음"으로 느껴진다. `null`과 실제 `<10`을 구분해야 한다.
- 자동 업데이트/NSIS는 `runAfterFinish:false`를 유지해야 하며, 업데이트 종료 흐름에서 앱이 의도치 않게 다시 뜨지 않는지 릴리스 전 수동 확인이 필요하다.
- 브라우저 자동화는 `puppeteer-pool` 이름을 쓰더라도 런타임은 Patchright 우선, Playwright fallback이어야 하며 종료 시 `closeIdle/destroy`가 호출되어야 한다.

## 릴리스 차단 조건

- `npm run verify:all` 실패 시 릴리스 금지.
- `scripts/run-sanity-gate-test.js`에 포함된 회귀 테스트 실패 시 릴리스 금지.
- 카테고리 기반 발굴 회귀 테스트는 다음을 보장한다.
  - 카테고리 단독 발굴 시 내부 시드 30개 이상 확보
  - `fashion`은 `fashion + beauty`로 유지
  - `쇼핑`은 `electronics + fashion + beauty`로 확장
  - 정책/건강 키워드가 카테고리 필터에서 탈락하지 않음
- SSS 등급은 `sanity-gate`를 통과한 데이터 기반 결과만 허용한다. 강제 승격, index 기반 SSS, 추정 SV/DC SSS는 금지.
- PRO 카테고리 모드는 기본적으로 watchlist 후보를 숨기지 않고 표시해야 한다. strict gate는 사용자가 명시적으로 켠 경우에만 차단 필터로 작동한다.

## 수동 QA 체크리스트

- 황금키워드 발굴
  - 정밀 모드에서 상위 SSR/SSS 후보가 30개 이상 노출되는지 확인
  - 6개 카테고리 프로필에서 카테고리별 후보가 한쪽으로 쏠리지 않는지 확인
  - 빅키워드 단일 토큰이 SSS 상단을 점령하지 않는지 확인
- PRO 트래픽 키워드 헌터
  - 카테고리 모드 50개 요청 시 30개 미만으로 줄지 않는지 확인
  - strict gate OFF에서 watchlist 후보가 경고 칩과 함께 유지되는지 확인
  - strict gate ON에서만 차단 개수가 늘어나는지 확인
- 마인드맵
  - 결과 모달이 항상 최상단으로 열리는지 확인
  - 검색량 미매칭은 `0`이 아니라 미측정/`<10` 의미로 표시되는지 확인
  - Seed 버튼에서 연관 마인드맵으로 이어지는 흐름이 끊기지 않는지 확인
- 정책브리핑/실시간
  - 정책브리핑 카드가 기사 본문 전체를 제목처럼 보여주지 않는지 확인
  - 지원금/복지/공고성 키워드가 실시간 후보로 들어오는지 확인
- 업데이트/프로세스
  - 설치 완료 후 앱 자동 실행이 꺼져 있는지 확인
  - 업데이트 재시작 흐름 외에 임의 relaunch가 없는지 확인
  - 종료 후 LEWORD 번들 Chromium 프로세스가 남지 않는지 확인

## 운영 원칙

- 기능 안내 문구와 실제 필터 동작이 다르면 코드가 회귀한 것으로 본다.
- "최소 30개"는 UI 표시 개수가 아니라 검증된 후보 반환 개수 기준이다.
- 대량 모드는 낮은 품질을 섞는 모드가 아니라 더 넓은 시드와 소스를 탐색하는 모드다.
- 추정값은 후보 판단 보조로만 쓰고, SSS/SS 등급 확정에는 실측 게이트를 통과시킨다.
- 카테고리 매핑은 `src/utils/category-discovery-map.ts`를 단일 진입점으로 유지한다.
